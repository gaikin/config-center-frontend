# 智能提示与智能作业运行记录查看设计

## 1. 背景

当前仓库中的“智能提示”和“智能作业”已经分别具备配置与运行的基础骨架：

- [PromptsPage.tsx](C:/dev/projects/work/config-center-frontend/src/pages/PromptsPage/PromptsPage.tsx) 承载智能提示入口，当前以规则配置为主。
- [RulesPage.tsx](C:/dev/projects/work/config-center-frontend/src/pages/RulesPage/RulesPage.tsx) 承载提示规则管理与编辑。
- [JobScenesPage.tsx](C:/dev/projects/work/config-center-frontend/src/pages/JobScenesPage/JobScenesPage.tsx) 已具备作业列表、作业编排和执行预览入口。
- [workflowService.ts](C:/dev/projects/work/config-center-frontend/src/services/workflowService.ts) 已具备部分执行记录与节点日志的 mock 数据能力。

但当前运行态可观测能力仍然偏弱，主要问题有：

- 智能提示缺少面向业务排查的命中记录列表入口。
- 智能作业缺少面向执行结果查询的统一运行记录入口。
- 业务页只能就近配置，无法在统一视图中按条件检索历史记录。
- 当前日志结构更偏原型或统计用途，缺少面向实际查询页的展示模型。

## 2. 目标与边界

### 2.1 目标

本次设计目标是补齐“查看记录”能力，让用户能快速回答以下问题：

- 最近命中过哪些提示规则。
- 某个页面、某个机构最近触发过哪些提示。
- 某个作业最近执行了多少次，结果如何。
- 哪些作业执行失败或部分成功，失败原因是什么。

本次能力定位为：`面向业务查询与轻量排障的记录中心`。

### 2.2 非目标

本次设计明确不做以下能力：

- 提示记录详情页或详情抽屉。
- 作业记录详情页或详情抽屉。
- 提示与作业的联查时间线。
- 全量节点级执行回放。
- 复杂审计分析、导出、订阅告警。
- 高级多条件组合查询。

## 3. 用户与核心场景

### 3.1 目标用户

首期用户主要是业务运营、实施交付和配置人员。

他们的主要诉求不是做深度技术排障，而是：

- 快速知道某条提示是否近期命中过。
- 快速知道某个作业是否执行失败。
- 按页面、机构、时间范围快速收敛问题范围。

### 3.2 核心路径

本次设计重点覆盖以下高频路径：

1. 从“智能提示”页进入命中记录，查看某条规则最近命中情况。
2. 从“智能作业”页进入运行记录，查看某个场景最近执行结果。
3. 从统一记录中心按名称、状态、时间、页面、机构筛选记录。
4. 根据列表中的失败原因摘要判断是否需要进一步回到配置页处理。

## 4. 方案选择

本次在以下三个方向中进行选择：

- 方案 A：只在业务页内增加记录 Tab。
- 方案 B：只做独立记录中心。
- 方案 C：业务页保留快捷入口，同时提供独立记录中心。

结论：采用 `方案 C`。

选择理由：

- 符合“业务页就近查看 + 统一中心全量检索”的使用习惯。
- 不会把记录能力分散在多个页面里，后续扩展成本更低。
- 能复用现有业务页上下文，减少用户重复筛选。

## 5. 信息架构

### 5.1 业务页快捷入口

智能提示页需要新增“命中记录”快捷入口：

- 页面级入口：查看全部提示命中记录。
- 规则上下文入口：自动带入规则筛选。

智能作业页需要新增“运行记录”快捷入口：

- 工具栏入口：查看全部作业运行记录。
- 场景行内入口：自动带入作业场景筛选。

业务页快捷入口统一跳转到记录中心，不在业务页内重复搭建一套完整记录视图。

### 5.2 运行记录中心

新增独立页面“运行记录中心”，内部采用双 Tab：

- `提示记录`
- `作业记录`

每个 Tab 共享同一套页面结构：

- 顶部筛选区
- 结果列表区
- 空状态和错误状态

本次不引入详情抽屉，因此页面只承担“筛选 + 列表展示”职责。

## 6. 页面与交互设计

### 6.1 提示记录

提示记录只记录“已命中”的提示规则，因此列表中不再展示“命中结果”列。

列表字段：

- 规则名
- 页面名称
- 所属机构
- 提示方式
- 触发时间
- 提示内容摘要
- 关联作业场景

筛选字段：

- 规则名称
- 时间范围，默认最近 7 天
- 页面
- 机构

交互规则：

- 从智能提示页进入时自动带规则筛选。
- 从页面上下文进入时自动带页面筛选。
- 点击规则名可回跳到对应规则配置页。
- 提示内容摘要过长时使用省略和悬浮展示完整文本。

### 6.2 作业记录

作业记录同样只做列表，不做详情。

列表字段：

- 作业名称
- 执行结果
- 页面名称
- 所属机构
- 触发来源
- 开始时间
- 执行耗时
- 失败原因摘要

筛选字段：

- 作业名称
- 执行结果
- 时间范围，默认最近 7 天
- 页面
- 机构

交互规则：

- 从智能作业页进入时自动带作业场景筛选。
- 成功、部分成功、失败记录都仅作为列表展示，不提供详情入口。
- 失败原因摘要应尽量业务化，例如“接口超时”“页面字段缺失”“脚本执行异常”。

### 6.3 排序与状态

两个 Tab 的列表默认统一：

- 按时间倒序。
- 首次进入默认最近 7 天。
- 筛选变化后即时刷新。

空状态文案建议直接反映当前条件，例如：

- “当前筛选条件下暂无提示命中记录”
- “当前筛选条件下暂无作业运行记录”

## 7. 数据模型设计

### 7.1 提示记录模型

新增面向展示的提示记录查询模型，字段控制在列表展示所需范围内：

```ts
interface PromptHitRecord {
  id: number;
  ruleId: number;
  ruleName: string;
  pageResourceId: number;
  pageResourceName: string;
  orgId: string;
  orgName: string;
  promptMode: "SILENT" | "FLOATING";
  promptContentSummary: string;
  sceneId?: number;
  sceneName?: string;
  triggerAt: string;
}
```

说明：

- 不增加 `hitResult` 字段，因为入表即表示命中。
- `promptContentSummary` 为面向列表的摘要字段，不直接承载完整富文本结构。

### 7.2 作业记录模型

作业记录首期可在现有 `JobExecutionSummary` 基础上补足展示所需字段：

```ts
interface JobExecutionRecord {
  id: number;
  sceneId: number;
  sceneName: string;
  pageResourceId: number;
  pageResourceName: string;
  orgId: string;
  orgName: string;
  triggerSource: "PROMPT_CONFIRM" | "FLOATING_BUTTON" | "AUTO" | "MANUAL_RETRY";
  result: "SUCCESS" | "PARTIAL_SUCCESS" | "FAILED";
  failureReasonSummary?: string;
  startedAt: string;
  finishedAt: string;
}
```

说明：

- `failureReasonSummary` 用于直接承载列表展示文案。
- 不再为首期设计额外详情模型。

## 8. 前端实现边界

### 8.1 页面层

首期前端需要补充：

- 新增“运行记录中心”页面。
- 在路由中挂载记录中心入口。
- 在菜单中增加独立入口，位置靠近智能提示和智能作业。
- 在智能提示页和智能作业页增加快捷跳转入口。

### 8.2 服务层

首期需要新增或补充以下查询接口：

- `listPromptHitRecords(filters)`
- `listJobExecutionRecords(filters)`

如果当前仍使用 mock/in-memory 数据，前端应先按最终展示模型组织 service 层返回值，避免后续切真实后端时页面层再改一遍。

### 8.3 URL 参数

为支持业务页快捷跳转，记录中心建议支持通过 URL 参数带默认筛选：

- `tab`
- `ruleId`
- `ruleName`
- `sceneId`
- `sceneName`
- `pageResourceId`

页面首次加载时仅将这些参数作为默认筛选值，不要求长期保留所有查询状态到 URL。

## 9. 错误处理与空态

列表查询失败时，统一提示：

- “记录加载失败，请稍后重试”

后端未返回失败原因时，作业记录列表统一降级显示：

- “暂无失败原因”

当页面、机构等筛选项不存在或已失效时，前端应保留当前值展示，并允许用户手动清空，避免因为历史数据脏值导致页面直接失真。

## 10. 测试建议

首期至少覆盖以下验证点：

- 记录中心双 Tab 切换与默认筛选生效。
- 从智能提示页跳转到提示记录时自动带入规则筛选。
- 从智能作业页跳转到作业记录时自动带入场景筛选。
- 提示记录列表不展示“命中结果”列。
- 作业记录列表不提供详情入口。
- 名称、时间、页面、机构、状态筛选生效。
- 查询失败和空状态文案正确展示。

## 12. 结论

本次设计采用“业务页快捷入口 + 独立记录中心”的双入口方案，但在能力上明确收敛为：

- 提示记录只做命中列表
- 作业记录只做执行结果列表
- 首期不做任何详情视图

这样可以以最小复杂度补齐运行记录查询能力，优先满足用户“看得到、查得到、定位得到”的基础诉求，并为后续是否追加详情能力保留扩展空间。
