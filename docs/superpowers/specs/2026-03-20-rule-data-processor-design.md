# 条件配置“预处理器”重构为“数据处理”设计

## 1. 背景

当前规则条件配置仍使用“预处理器”概念，且执行层存在按固定 ID（3501/3502/3503）分支处理的内置逻辑。该实现与“统一标准 JS 函数”目标冲突，也会增加后续扩展维护成本。

本次改造目标是：将条件配置中的预处理能力统一升级为“数据处理”，全部采用标准 JS 函数并通过统一调用协议执行。

## 2. 已确认决策

- 采用全量重构方案，不兼容旧内置逻辑。
- 函数定义格式固定为标准函数声明：`function transform(...) { ... }`。
- 第 1 个参数固定接收外部输入 `input`。
- 函数定义时显式配置总参数个数 `paramCount = n`（`n >= 1`）。
- 条件配置侧根据 `paramCount` 渲染 `n-1` 个参数输入框。
- 组织范围不再可配置，默认“所有人可见”（前端隐藏）。

## 3. 方案对比

### 方案 A：全量重构（采用）

- 统一替换命名、类型、UI 文案与执行逻辑。
- 清理旧内置分支与相关 seed。
- 优点：语义一致、实现干净、后续扩展成本最低。
- 缺点：一次性改动面更大。

### 方案 B：UI 改名，内部保留兼容

- 页面显示“数据处理”，内部仍沿用 preprocessor 语义与部分旧字段。
- 优点：短期改动小。
- 缺点：技术债保留，后续仍需二次重构。

### 方案 C：并行新增模块

- 新增 data-processor 模块，与旧 preprocessor 并行。
- 优点：迁移可分阶段。
- 缺点：链路复杂，短期维护负担上升。

## 4. 实施设计

### 4.1 类型与数据模型

- `PreprocessorDefinition` 重命名为 `DataProcessorDefinition`。
- 新结构：
  - `id`
  - `name`
  - `category`
  - `paramCount`
  - `functionCode`
  - `status`
  - `usedByCount`
  - `updatedAt`
- 删除 `processorType` 与 `ownerOrgId` 的前台维护含义。

规则操作数结构同步重命名：

- `preprocessorIds` -> `dataProcessorIds`
- `preprocessorConfigs` -> `dataProcessorConfigs`
- 每项配置结构：`{ dataProcessorId: number; args: string[] }`

### 4.2 高级配置与数据处理管理页

- 高级配置 Tab 名称改为“数据处理”。
- 管理页标题改为“数据处理函数”。
- 列表字段：名称、分类、被引用次数、状态、更新时间、操作。
- 表单字段：名称、分类、参数个数、函数代码、状态。
- 组织范围字段隐藏，不允许用户编辑。
- 新建默认模板根据 `paramCount` 生成：

```js
function transform(input, arg1, arg2) {
  return input;
}
```

（示例为 `paramCount=3`）

校验规则：

- `paramCount >= 1`
- `functionCode` 必须包含 `function transform(` 声明
- `functionCode` 的形参数量必须与 `paramCount` 一致

### 4.3 规则条件配置页

- 文案统一替换：
  - “预处理器” -> “数据处理”
  - “添加预处理器” -> “添加数据处理”
- 在每个操作数属性面板中保留“数据处理链”。
- 每个数据处理项渲染：
  - 数据处理函数选择器
  - 动态参数输入区（`n-1` 个输入框）
- 切换函数时，参数输入区按目标函数 `paramCount` 重新渲染。
- 来源切换提示文案改为会清空“数据处理配置”。

### 4.4 执行引擎

在 `workflowService` 中重构执行逻辑：

- 删除基于 3501/3502/3503 的硬编码分支。
- 新增通用函数执行器：
  - 从 `functionCode` 构造并获取 `transform`
  - 以 `transform(input, ...args)` 调用
- 多个数据处理函数按配置顺序串行执行：
  - 上一个函数返回值作为下一个函数 `input`

运行异常策略：

- 任一函数执行失败时记录 trace；当前条件按不命中处理。
- 规则页保存阶段阻断“引用了不存在的数据处理函数”的配置。

### 4.5 数据与种子清理

- 删除旧内置数据（3501/3502/3503）与旧文案。
- 既有规则中的旧预处理引用不做兼容转换，按“无效引用”校验阻断并提示用户重新选择数据处理函数。

## 5. 风险与缓解

- 风险：历史规则存在旧引用，改造后可能无法直接保存。
- 缓解：在条件配置面板提供明确阻断提示，指导用户重新绑定数据处理函数并补齐参数。

- 风险：函数代码可执行带来运行不确定性。
- 缓解：保存时做基础语法与签名校验；运行时捕获异常并显示可读错误。

## 6. 测试计划

### 6.1 服务层

- 新增/编辑数据处理函数时参数个数校验。
- 启停状态与列表查询正常。

### 6.2 规则页

- 选择函数后正确渲染 `n-1` 参数输入框。
- 切换函数时参数框数量及时刷新。
- 无效函数引用触发阻断校验。

### 6.3 执行层

- `transform(input, ...args)` 正常执行并串联。
- 运行报错时不中断页面，返回可见错误信息。

### 6.4 回归

- 高级配置页 Tab 与文案一致。
- 条件配置保存/切换无闪烁报错。
- `npm run build` 通过。
