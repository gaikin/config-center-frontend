# 外部主数据引用与页面能力绑定设计

## 1. 背景

当前系统正在从“主数据管理型配置中心”收敛为“关系与能力配置中心”。

结合现有约束，本系统不再维护以下主数据：

- 菜单主数据
- 页面主数据
- 业务流程主数据

这些对象统一由外部系统提供，本系统只通过查询接口读取。

在这个前提下，系统要解决的核心问题变成了：

- 如何基于 `menuCode` 控制智能提示和智能作业的开通范围
- 如何把智能提示和智能作业挂到具体页面上
- 如何把任务中心中的 `业务流程` 组织成固定顺序的页面链路
- 如何保证这些配置可见、可查、可审计

## 2. 目标与边界

### 2.1 目标

本次设计目标是建立一套清晰的引用与绑定模型，让系统可以：

- 查询外部系统返回的菜单、页面、业务流程信息
- 基于 `menuCode` 管理菜单能力开通
- 基于页面资源管理智能提示和智能作业的挂载关系
- 基于业务流程管理一串固定顺序的页面步骤
- 保留完整的配置审计和变更记录

### 2.2 非目标

本次明确不做以下内容：

- 菜单主数据维护
- 页面主数据维护
- 业务流程主数据维护
- 菜单树、页面树、流程树的创建与编辑
- 外部系统数据的同步任务管理
- 运行态详情回放、复杂分析报表

## 3. 方案选择

本次在三种结构中选择 `方案 A`：

- 方案 A：外部主数据为准，本系统只做关系和能力配置
- 方案 B：外部主数据为准，本系统同时做缓存镜像
- 方案 C：本系统维护主数据，外部仅作参考

结论：采用 `方案 A`。

选择理由：

- 与当前边界最一致
- 不重复维护菜单、页面、业务流程主数据
- 实现成本最低
- 后续如果外部系统调整字段，只需要适配查询接口

## 4. 总体模型

### 4.1 主体对象

系统中涉及的对象分为两类：

#### 外部查询对象

- 菜单
- 页面
- 业务流程

这些对象不在本系统落库，仅通过外部接口查询。

#### 本系统配置对象

- 菜单能力开通记录
- 页面能力绑定记录
- 业务流程步骤配置
- 审计记录
- 变更记录

### 4.2 语义边界

- `menuCode` 只用于普通菜单识别和能力开通范围判断
- 智能提示和智能作业最终都挂到具体页面上
- 业务流程是任务中心的特殊模块，与菜单主数据无直接依赖
- 业务流程由一串固定顺序的页面组成
- 本系统只保存“引用关系”和“挂载关系”，不保存主数据本体

## 5. 菜单能力开通

### 5.1 目标

菜单能力开通用于控制某个菜单是否可以启用：

- 智能提示
- 智能作业

这里的识别维度是 `menuCode`。

### 5.2 数据结构

建议本系统保存以下信息：

- `menuCode`
- 能力类型：`PROMPT` / `JOB`
- 开通状态：`ENABLED` / `DISABLED` / `PENDING`
- 生效范围
- 更新时间
- 更新人
- 审计信息

### 5.3 行为规则

- 菜单是否开通，只影响该菜单下页面是否允许挂载能力
- 菜单开通不等于页面自动绑定
- 菜单开通后，页面仍需显式绑定智能提示或智能作业

### 5.4 展示方式

菜单能力开通建议在菜单管理或能力管理页中展示为：

- 菜单名称
- `menuCode`
- 提示能力状态
- 作业能力状态
- 当前可用页面数
- 最近更新时间

## 6. 页面能力绑定

### 6.1 目标

页面能力绑定用于管理智能提示和智能作业最终挂载到哪个具体页面上。

### 6.2 数据结构

建议每条页面绑定记录包含：

- 页面引用 ID
- 页面名称快照或显示名
- 能力类型：`PROMPT` / `JOB`
- 能力引用 ID
- 启用状态
- 优先级
- 排序值
- 更新时间
- 更新人

### 6.3 规则

- 一个页面可以绑定多个智能提示
- 一个页面可以绑定多个智能作业
- 同一能力可以绑定多个页面
- 页面绑定必须显式配置，不默认继承整个菜单
- 页面仅在菜单已开通相应能力后，才允许新增绑定

### 6.4 交互建议

页面绑定应以“页面详情”为中心：

- 在页面详情里查看当前页已绑定的提示和作业
- 允许新增、启停、调整优先级、解绑
- 不在主数据层维护页面，只通过外部查询选择页面

## 7. 业务流程

### 7.1 定义

业务流程是任务中心的特殊对象，用于描述一串固定顺序的页面步骤。

业务流程不是菜单，不依赖菜单树，也不由菜单主数据驱动。

### 7.2 结构

一个业务流程由以下部分组成：

- 业务流程标识
- 流程名称
- 页面步骤列表
- 每一步的顺序号
- 每一步的页面引用
- 每一步挂载的智能提示和智能作业
- 流程状态

### 7.3 约束

- 流程步骤必须按固定顺序串联
- 同一个业务流程中的页面必须属于同一菜单来源
- 页面步骤不能跨菜单拼接
- 每一步必须显式指定页面
- 业务流程本身不保存页面主数据，只保存页面引用

### 7.4 运行方式

运行时按步骤顺序推进：

1. 进入业务流程
2. 命中第一个页面步骤
3. 加载该页面下已绑定的提示和作业
4. 完成后推进到下一步页面
5. 直到流程结束

## 8. 权限与可见性

### 8.1 菜单能力开通权限

菜单能力开通建议遵循现有配置权限体系：

- 可查看的人可以看到开通状态
- 可配置的人可以修改开通状态
- 高权限人员可以处理特殊菜单或特殊流程

### 8.2 页面绑定权限

页面绑定建议分两层控制：

- 能看到页面的人，才能看到页面绑定结果
- 能配置页面能力的人，才能新增或修改绑定

### 8.3 业务流程权限

业务流程建议独立配置权限：

- 业务流程查看
- 业务流程编辑
- 业务流程启停
- 业务流程复制或另存

## 9. 查询与展示

### 9.1 外部查询

页面上展示的信息都来自外部系统查询：

- 菜单列表
- 页面列表
- 业务流程列表

本系统只在查询结果上叠加：

- 能力开通状态
- 页面绑定结果
- 流程步骤配置

### 9.2 展示层建议

建议把界面拆成三块：

- 菜单能力开通
- 页面能力绑定
- 业务流程配置

这样做的原因是：

- 菜单是入口层
- 页面是挂载层
- 业务流程是编排层

## 10. 异常与校验

### 10.1 外部查询异常

如果外部系统查询失败：

- 菜单/页面/业务流程列表应给出明确失败提示
- 已加载的绑定配置可以保留，但需标记外部数据不可用
- 不允许基于过期主数据继续新增绑定

### 10.2 保存校验

保存绑定时应校验：

- 菜单是否已开通对应能力
- 页面是否存在
- 业务流程步骤是否完整
- 流程页面是否属于同一菜单来源
- 页面顺序是否重复或断裂

### 10.3 删除与失效

如果外部系统删除了页面或菜单：

- 本系统保留历史绑定记录
- 相关配置应显示“引用对象已失效”
- 不允许对失效对象继续编辑或新增绑定

## 11. 前端实现边界

前端首期建议只实现以下内容：

- 菜单能力开通页
- 页面能力绑定页
- 业务流程配置页
- 外部对象查询与选择
- 绑定关系展示
- 启停和审计入口

前端不实现：

- 菜单树维护
- 页面树维护
- 业务流程主数据维护

## 12. 后端实现边界

后端首期建议提供：

- 外部对象查询适配层
- 菜单能力开通 CRUD
- 页面绑定 CRUD
- 业务流程配置 CRUD
- 审计记录查询

后端不需要：

- 主数据同步任务
- 主数据编辑接口
- 复杂编排引擎

## 13. 成功标准

本次设计落地后，以下闭环需要成立：

1. 菜单、页面、业务流程都来自外部查询。
2. 本系统不维护这些主数据。
3. 菜单可按 `menuCode` 开通提示/作业能力。
4. 页面可绑定多个提示和多个作业。
5. 业务流程可按固定顺序串联页面。
6. 所有绑定关系可查询、可启停、可审计。

## 14. 结论

本次采用“外部主数据引用 + 本系统关系配置”的方案。

这意味着：

- 菜单主数据不落本地
- 页面主数据不落本地
- 业务流程主数据不落本地
- 本系统只管开通、绑定、编排和审计

最终形成的模型是：

- `menuCode` 负责菜单级能力开关
- 页面负责能力挂载点
- 业务流程负责固定顺序的页面链路

这个边界最适合当前系统定位，也最容易继续往后扩展。
